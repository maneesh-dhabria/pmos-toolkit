# /prototype Skill — Design Spec

**Date:** 2026-04-30
**Status:** Design approved, ready for implementation planning
**Pipeline:** pmos-toolkit

---

## Purpose

Optional bridge between `/wireframes` and `/spec` that produces a high-fidelity, interactive, single-HTML-per-device prototype so stakeholders can experience a feature end-to-end before committing to implementation. Mock data is LLM-generated from the requirements doc to feel domain-real. The prototype stitches all wireframe screens into walkable journeys with simulated API calls, forms, CRUD, and loading/error states.

The prototype is **upstream** of `/spec` — it is a stakeholder-confidence and design-validation artifact, not the implementation. The single-HTML-per-device constraint is the scope fence that keeps it from drifting into `/execute` territory.

---

## Pipeline Position

```
/requirements → /wireframes → [/prototype] → /spec → /plan → /execute → /verify
                              optional bridge
```

- Auto-triggers `/wireframes` if no wireframes folder exists for the feature
- Auto-triggers `/requirements` (via `/wireframes`) if no req doc exists

---

## Tier Gating

Tier is read from the requirements doc tag (carried forward from `/requirements`). If absent, the skill asks via `AskUserQuestion`.

| Tier | Description | /prototype behavior |
|------|-------------|---------------------|
| Tier 1 | Bug fix / minor enhancement | **Skip** — recommend going straight to `/spec` |
| Tier 2 | Enhancement / UX overhaul | **Optional** — gated via `AskUserQuestion` |
| Tier 3 | New feature / major redesign | **Mandatory** — announce and run |

---

## Inputs

1. **Requirements doc** — journeys, business rules, entity model, tier tag
2. **Wireframes folder** — visual reference, IA, copy, state matrix, `house-style.json`
3. **All wireframe screens** — full coverage; no scope cherry-picking. Devices default to the wireframes' device list.

---

## Output Structure

```
{feature_folder}/prototype/
  index.html                    # landing page with device tabs
  index.desktop-web.html        # one file per device
  index.mobile-web.html
  index.ios-app.html
  assets/
    runtime.js                  # SHARED: router, mock-API client, in-memory store, latency simulator
    components.js               # SHARED: atoms (Button, Input, Modal, Toast, …)
    styles.css                  # high-fidelity skin (derived from wireframes' house-style.json)
    mock-data.json              # entity summary / index
    users.json                  # per-entity mock data
    products.json
    …
    inline-data.html            # <script type="application/json"> fallback for file:// portability
  prototype-findings.md         # review-loop findings (Phase 6)
  interactive-friction.md       # clicks/keystrokes/decisions per journey (Phase 7)
```

`runtime.js` and `components.js` are **shared across all device files** to avoid duplicating ~500 lines of plumbing per device.

---

## Tech Stack (per device file)

- **React 18 + ReactDOM** via unpkg CDN
- **Babel Standalone** for in-browser JSX compilation
- **JSX** authoring style (no build step, no npm packages, no bundler)
- One inline `<script type="text/babel">` per concern (runtime hookup / screens / app shell)
- `assets/runtime.js` and `assets/components.js` loaded as plain JS modules
- **High-fidelity styling**: real brand colors and typography pulled from wireframes' `house-style.json`. No wireframe annotation chrome.
- **`file://` portability**: `fetch('./assets/users.json')` with a fallback that reads inline `<script type="application/json" id="mock-users">…</script>` blocks if `fetch` fails (Chrome/Safari block `fetch` from `file://` by default). Generator emits both.
- **No backend, no build step, no npm install required.** The user can email the folder to a stakeholder and it works.

---

## Interactivity Baseline

Every prototype MUST simulate:

1. **Client-side navigation** between all wireframe screens (hash router or in-memory)
2. **Forms with real validation + mock submit** (success/error decided by mock-data rules; simulated 200–800ms latency)
3. **CRUD-ish actions** (create/edit/delete) — changes persist in the in-memory store for the session, lost on reload
4. **Loading / error / empty states** triggered by the simulated-latency layer; some routes deterministically fail to demonstrate error recovery

---

## Mock Data Strategy

- **Source of truth**: separate JSON files under `assets/` (`users.json`, `products.json`, etc.)
- **Generation**: LLM-generated by the skill from `requirements doc + wireframes`. Domain-appropriate names, prices, dates, copy. No Lorem ipsum, no "User 1 / Item A".
- **Volume**: enough records to make the experience feel lived-in (typically 20–200 per entity depending on the screen). The skill chooses a reasonable default per entity based on what the wireframes display (e.g., a list view showing 10 rows → generate 50–100 records so pagination/filtering feels real).
- **Generated once per session**. Not regenerated per device file.
- **Inlined as fallback**: generator also emits `<script type="application/json" id="mock-{entity}">` blocks in each device file so `file://` opens still work.
- **No runtime faker, no schema regen subcommand** — keeping scope tight.

## Phases

### Phase 0: Load Workstream Context & Learnings
- Follow `product-context/context-loading.md`
- Read `~/.pmos/learnings.md` entries under `## /prototype`
- Resolve feature folder via `_shared/feature-folder.md`

### Phase 1: Locate Inputs
- Find requirements doc; if missing, trigger `/requirements` then `/wireframes`
- Find wireframes folder; if missing, trigger `/wireframes`
- Read tier tag from req doc

### Phase 2: Tier Gate & Scope Confirmation
- Tier 1 → exit with recommendation to run `/spec`
- Tier 2 → `AskUserQuestion` opt-in
- Tier 3 → announce mandatory and proceed
- Confirm device list (default = wireframes' devices)
- Confirm interactivity baseline (default = all four types on)

### Phase 3: Generate Mock Data
- Read req doc + wireframes to identify entities and visible fields
- LLM-generate domain-appropriate JSON per entity
- Write to `assets/<entity>.json` and `assets/mock-data.json` (index)
- User reviews via `AskUserQuestion`: **Use as generated** / **Edit before continuing** / **Regenerate**

### Phase 4: Generate Shared Runtime + Components
- Generate `assets/runtime.js` (router, mock-API client with latency simulator, in-memory store, fetch+inline-fallback loader)
- Generate `assets/components.js` (Button, Input, Modal, Toast, Card, Table — atoms shared across screens)
- Generate `assets/styles.css` (high-fi skin from wireframes' `house-style.json`)
- Subagents in parallel where available

### Phase 5: Generate Per-Device HTML Files
- One subagent per device file (each subagent generates ALL screens for its device to keep them visually consistent)
- Each device file: HTML shell + `<script type="text/babel">` blocks for screens and app
- Loads shared `assets/runtime.js`, `assets/components.js`, `assets/styles.css`
- Embeds `<script type="application/json">` fallbacks for every entity JSON
- Implements navigation across all wireframe screens
- Wires up forms, CRUD actions, loading/error/empty states using the runtime layer

### Phase 6: Refinement Loop (Reviewer Subagent + ≤2 Loops Per File)
- Per device file: dispatch reviewer subagent
- **Rubric** (`reference/eval-rubric.md`): interactivity completeness, journey coverage, mock-data realism, accessibility, visual consistency, runtime errors (no console errors on first paint or on any navigation step)
- High/medium findings → apply fixes; low findings → log
- Hard cap: 2 loops per file
- Cross-file rollup of unresolved high/medium findings via Findings Presentation Protocol

### Phase 7: Interactive Friction Pass
- For each user journey (cap 5):
  - Trace screens in order through the running prototype (subagent walks the journey)
  - Count clicks, keystrokes, decisions per step
  - Flag flows exceeding thresholds (e.g., >12 clicks for first-value, >3 form steps to complete primary action)
- Output: `interactive-friction.md` with per-journey rollup
- Not a re-run of PSYCH/MSF — those stay in `/wireframes`

### Phase 8: Findings Presentation Protocol
- Group findings (review loop + friction pass) by target
- `AskUserQuestion`, ≤4 per batch
- Per-finding options: **Apply to prototype** / **Update wireframe** / **Update req doc** / **Defer to spec**
- Apply dispositions; log all changes in `prototype-findings.md`
- Platform fallback: numbered findings table with disposition column

### Phase 9: Generate Landing Index + Serve
- `prototype/index.html`: device tabs linking to per-device prototypes; mock-data summary; finding counts
- Detect Node; if available, run `npx http-server` and report URL
- Always print absolute `file://` path as fallback (with note about Chrome/Safari fetch restrictions; inline-data fallback should make this work anyway)

### Phase 10: Spec Handoff
- Append to requirements doc:

  ```markdown
  ## Prototype

  Generated: {YYYY-MM-DD}
  Folder: `{relative_path_to_prototype}`
  Index: `{relative_path}/index.html`
  Devices: desktop-web, mobile-web, ios-app
  Mock data: 4 entities, ~200 records
  Findings: `{relative_path}/prototype-findings.md`
  Friction pass: `{relative_path}/interactive-friction.md`
  ```

- Commit: `docs: add prototype for <feature>`

### Phase 11: Workstream Enrichment
- If workstream loaded in Phase 0, follow `product-context/context-loading.md` Step 4
- Signals: brand color overrides made during Phase 4, novel interaction patterns, device-specific decisions

### Phase 12: Capture Learnings
- Run `learnings/learnings-capture.md`
- Reflect on session; propose entries for `~/.pmos/learnings.md` under `## /prototype`

---

## Findings Presentation Protocol

(Same shape `/wireframes` uses — keep consistent.)

1. Group findings by category (max 4 per batch)
2. One question per finding via `AskUserQuestion`:
   - `question`: one-sentence finding + which file(s) + proposed fix
   - `options`: **Apply to prototype** / **Update wireframe** / **Update req doc** / **Defer to spec**
3. Batch ≤4 per call; sequential calls for more
4. Open-ended findings → inline follow-up
5. Platform fallback: numbered findings table with disposition column; do NOT silently self-fix
6. **Anti-pattern**: a wall of prose ending in "let me know what to fix"

---

## Anti-Patterns

- DO NOT generate prototype without confirmed wireframes (auto-trigger `/wireframes` if missing)
- DO NOT regenerate mock data per device file — generate once, share across all
- DO NOT silently fix high-severity findings — always surface via Findings Presentation Protocol
- DO NOT exceed 2 refinement loops per device file
- DO NOT prototype non-user-facing features (backend-only, cron jobs, internal APIs) — recommend skipping
- DO NOT introduce build steps, npm packages, bundlers, or any backend
- DO NOT inline `runtime.js` into device files — share it via `assets/`
- DO NOT use Lorem ipsum or generic mock data ("User 1", "Item A") — must be domain-real
- DO NOT cherry-pick screens — full coverage is the contract; if the user wants partial, they should run `/wireframes` with a narrower scope first
- DO NOT re-run PSYCH or MSF — those are `/wireframes` responsibilities; `/prototype` runs the lighter interactive-friction pass instead
- DO NOT skip Phase 12 (capture learnings) — terminal gate
- DO NOT generate prototypes that fail on `file://` — always emit the inline-data fallback alongside JSON files
- DO NOT exceed 5 journeys in the friction pass — diminishing returns
- DO NOT auto-edit upstream wireframes or req doc without explicit user disposition
